# 短链接项目：智能投放与分析 Agent 正式版开发文档（讨论稿）

> 文档性质：Codex 后续开发执行契约草案  
> 适用对象：智能投放与分析 Agent 的后续实现、测试、集成和验收  
> 当前任务边界：本文档只定义开发计划，不代表当前代码已经实现；不得据此跳过用户确认直接改代码  
> 实施原则：Agent 先独立构建，再通过 `admin` 扩展接入；不完全重构原短链接项目  

---

## 0. 文档定位

本文档用于指导后续 Codex 分阶段开发“智能投放与分析 Agent”。

第一阶段目标不是重写短链系统，而是在现有系统旁边构建一个可独立测试的 Agent 能力层：

```text
Agent Service 负责自然语言和编排；
Admin Adapter 负责权限、工具、安全和执行；
Project 保持现有短链与统计事实源职责；
Gateway/Nginx 只做必要路由接入。
```

---

## 1. 建设目标

### 1.1 产品目标

让后台用户可以用自然语言完成以下工作：

```text
帮我为“暑期活动”生成 5 条渠道短链；
把这个落地页做成短链，有效期到月底；
分析最近 7 天哪个分组表现最好；
这个短链为什么昨天点击突然变多；
给我总结一个活动复盘，说明地区、设备和时间段表现；
根据已有数据建议我下一步应该优化什么。
```

### 1.2 工程目标

1. Agent 运行逻辑与原业务服务解耦。
2. Agent 工具只通过受控 Adapter 访问短链能力。
3. 读操作可直接执行，写操作必须确认。
4. 所有 Agent 回答可追溯到工具结果。
5. 不影响现有短链创建、跳转和统计主链路。

### 1.3 成功标准

第一阶段完成后，至少支持：

| 场景 | 验收标准 |
---|---|
| 查询活动表现 | Agent 能调用分组统计并返回真实 PV/UV/UIP、趋势和建议 |
| 查询单链表现 | Agent 能调用单链统计并解释访问画像 |
| 创建单条短链 | Agent 能生成创建草案，用户确认后创建 |
| 批量创建短链 | Agent 能按渠道列表生成批量创建草案，用户确认后执行 |
| 异常解释 | Agent 能基于 Top IP、UV/PV 比、设备/地区集中度给出可能原因 |
| 权限隔离 | 用户不能通过 Agent 查询不属于自己的 gid |
| 写操作安全 | 未确认的写操作不会执行 |

职责：

```text
封装 ShortLinkActualRemoteService；
```

建议包结构：

```text

  service/
  dto/req/
  dto/resp/
```

#### 2.1.3 Agent 会话和动作确认

必须支持：

```text
session；
message；
tool call；
pending action；
action confirm；

```

#### 2.1.4 只读分析工具

必须封装：



## 3. 模块设计

### 3.1 Agent API 模块

#### 模块职责

接收用户对话请求，返回 Agent 回复。

#### API

```text
POST /internal/short-link-agent/v1/chat
GET  /internal/short-link-agent/v1/health
```

#### 输入

```json
{
  "sessionId": "optional",
  "username": "from-admin-not-from-browser",
  "message": "帮我分析最近 7 天暑期活动表现",
  "context": {
    "source": "admin",
    "locale": "zh-CN"
  }
}
```

#### 输出

```json
{
  "sessionId": "agt_xxx",
  "answer": "最近 7 天...",
  "cards": [],
  "pendingActions": [],
  "dataSources": [],
  "warnings": []
}
```

#### 禁止事项

```text
不得信任前端 username；
不得直接访问 MySQL/Redis；
不得生成未绑定工具结果的数字指标；
不得自动执行写工具。
```

### 3.2 Intent Router

#### 识别意图

| 意图 | 说明 |
|---|---|
| `CREATE_LINK` | 创建单条短链 |
| `BATCH_CREATE_LINKS` | 批量创建短链 |
| `ANALYZE_GROUP` | 分析分组/活动表现 |
| `ANALYZE_LINK` | 分析单条短链 |
| `EXPLAIN_ANOMALY` | 异常解释 |
| `GENERATE_REPORT` | 生成复盘报告 |
| `CLARIFY` | 参数缺失追问 |
| `GENERAL_HELP` | 使用帮助 |

#### 关键要求

1. 缺少 `gid` 时优先调用 `list_groups` 或追问。
2. 缺少时间范围时默认最近 7 天，并在回答中说明。
3. 涉及创建时必须生成确认草案。
4. 涉及删除/回收时第一阶段拒绝并说明原因。

### 3.3 Tool Gateway Client

#### 职责

Agent Service 中的工具客户端，只能调用 Admin Adapter 暴露的工具 API。

#### 重试策略

| 错误 | 策略 |
---|---|
| 网络超时 | 最多重试 1 次 |
| 401/403 | 不重试，返回权限错误 |
| 业务参数错误 | 不重试，触发追问 |
| 统计无数据 | 不报错，返回空状态 |
| 写操作失败 | 返回失败原因，不自动重放 |

### 3.4 Analysis Engine

#### 职责

所有数值分析必须由确定性代码完成：

```text
排序；
聚合；
环比；
占比；
Top N；
异常初筛；
UV/PV 比；
UIP/PV 比；
设备/地域集中度；
时间段峰值；
新老访客结构。
```

LLM 只负责：

```text
把确定性分析结果解释成人话；
生成建议；
组织报告结构；
追问缺失参数。
```

### 3.5 Response Composer

输出必须包含：

```text
一句话结论；
关键指标；
证据或数据来源；
可能原因；
建议动作；
风险提示；
后续可执行操作。
```

示例：

```text
结论：最近 7 天“暑期活动”主要流量来自移动端，UV 集中在周三和周五晚间。
依据：get_group_stats 返回 2026-07-01 至 2026-07-07 的 daily/hour/device 数据。
建议：下次投放优先安排在 19:00-22:00，并单独生成移动端渠道短链。
```

## 4. Admin Adapter 接口设计

### 4.1 Chat Proxy

可选：前端先调 admin，再由 admin 调 Agent Service。

```text
Method: POST
Path: /api/short-link/admin/v1/agent/chat
Auth: gateway token
```

职责：

```text
读取 UserContext；
把 username/userId 安全传给 Agent Service；
不把 token 传给 Agent；
返回 Agent 结构化响应。
```

### 4.2 Tool API

内部 API 示例：

```text
POST /internal/admin-agent-tools/list-groups
POST /internal/admin-agent-tools/page-short-links
POST /internal/admin-agent-tools/get-group-stats
POST /internal/admin-agent-tools/create-pending-action
POST /internal/admin-agent-tools/execute-pending-action
```

所有工具 API 必须：

```text
接收 internal auth；
接收 username；
校验 gid；
脱敏输出；
记录 audit；
限制分页和批量数量。
```

### 4.3 PendingAction API

```text
POST /api/short-link/admin/v1/agent/actions/{actionId}/confirm
POST /api/short-link/admin/v1/agent/actions/{actionId}/cancel
```

执行规则：

```text
action 必须属于当前 username；
action 必须是 pending 状态；
action 未过期；
payload hash 未变化；
确认后只能执行一次；
执行结果必须写审计。
```

## 5. 数据库设计

### 5.1 第一阶段表清单

建议在 `admin` 数据源所在库中新增单表：

```text
t_agent_session
t_agent_message
t_agent_tool_call
t_agent_pending_action
```

第一阶段不分表。后续如果会话量变大，再评估按 `username` 或 `session_id` 分表。

### 5.2 字段规范

所有 Agent 表必须包含：

```text
id；
create_time；
update_time；
del_flag。
```

涉及用户的数据必须包含：

```text
username；
user_id 可选；
trace_id；
session_id。
```

### 5.3 数据保留

建议：

| 数据 | 默认保留 |
---|---|
| session/message | 90 天 |
| tool_call | 30 天 |
| pending_action | 180 天 |
| 错误日志 | 30 天 |

> 待确认事项：如果用于课程演示，可先不做自动清理，但表结构应预留。

## 6. 配置设计

### 6.1 Agent Service 环境变量

```text
AGENT_SERVICE_PORT=8010
SHORTLINK_ADMIN_BASE_URL=http://127.0.0.1:8002
AGENT_INTERNAL_TOKEN=
LLM_PROVIDER=openai-compatible
LLM_BASE_URL=
LLM_API_KEY=
LLM_MODEL=
AGENT_TOOL_TIMEOUT_MS=5000
AGENT_MAX_BATCH_CREATE_SIZE=20
AGENT_TRACE_ENABLED=true
```

### 6.2 Admin 配置

```yaml
short-link:
  agent:
    enabled: true
    service-url: http://127.0.0.1:8010
    internal-token: ${AGENT_INTERNAL_TOKEN:}
    max-batch-create-size: 20
    tool-timeout-ms: 5000
```

### 6.3 Feature Flag

| 配置 | 默认 | 说明 |
---|---|---|
| `short-link.agent.enabled` | false | 是否启用 Agent API |
| `short-link.agent.write-enabled` | false | 是否允许确认后写操作 |
| `short-link.agent.risk-scan-enabled` | false | 是否启用 URL 风险扫描 |
| `short-link.agent.trace-enabled` | true | 是否记录工具轨迹 |

## 9. Codex 开发阶段拆解

### Phase 1：文档确认与边界冻结

#### 目标

确认第一阶段只做智能投放与分析 Agent，不做安全风控实装和智能路由。

#### 输出

```text
最终版计划文档；
确认技术栈；
确认 Agent 服务形态；
确认写操作策略。
```

#### 禁止事项

```text
禁止直接开始改业务代码；
禁止扩大到广告平台；
禁止改跳转主链路。
```

### Phase 2：Agent Service 骨架

#### 目标

创建独立 Agent 服务骨架，支持健康检查、chat API、工具客户端 mock。

#### 涉及文件

```text
agent-service/**
```

#### 验收标准

```text
服务可启动；
health API 返回 OK；
chat API 可返回 mock；
单元测试通过。
```

### Phase 3：Admin Adapter 只读工具

#### 目标

新增 admin 侧内部工具 API，封装分组、短链列表、统计查询。

#### 涉及文件

```text
admin/src/main/java/com/nageoffer/shortlink/admin/agent/**
admin/src/main/resources/application.yaml
```

#### 验收标准

```text
工具 API 能校验当前用户；
gid 越权被拒绝；
统计结果可脱敏返回；
已有后台接口不受影响。
```

### Phase 4：Agent 分析闭环

#### 目标

Agent 能调用真实只读工具，完成分组/单链分析。

#### 验收标准

```text
回答包含真实数据来源；
空数据场景不报错；
工具失败有降级回答；
生成结构化 cards。
```

### Phase 5：PendingAction 与写操作确认

#### 目标

支持创建分组、创建短链、批量创建短链的确认执行。

#### 验收标准

```text
未确认不执行；
确认后只执行一次；
过期动作不可执行；
批量数量受限；
执行结果可追溯。
```

### Phase 6：网关与前端接入

#### 目标

通过现有后台入口接入 Agent。

#### 验收标准

```text
Gateway/Nginx 路由可达；
登录用户可使用 Agent；
未登录用户不可访问；
前端展示回答、指标卡、PendingAction。
```

### Phase 7：回归、验收与第二阶段评审

#### 目标

完成回归测试和验收，并决定安全风控 Agent 的实现计划。

#### 验收标准

```text
现有短链创建/跳转/统计不回归；
Agent 场景测试通过；
验收清单有证据；
第二阶段边界明确。
```

## 10. 开发红线

1. 不允许 Agent 直接访问业务数据库。
2. 不允许 Agent 绕过 admin 权限。
3. 不允许 Agent 自动执行删除、回收、更新跳转目标。
4. 不允许 LLM 输出未经工具证实的指标。
5. 不允许把用户 token、手机号、邮箱、完整 IP 注入模型。
6. 不允许在跳转热路径调用大模型。
7. 不允许为了 Agent 第一阶段重构所有统计链路。
8. 不允许把安全风控、智能路由、转化归因混入第一阶段验收。

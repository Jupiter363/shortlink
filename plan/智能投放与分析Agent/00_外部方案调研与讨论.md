# 短链接项目：智能投放与分析 Agent 外部方案调研与讨论稿

> 文档性质：外部方案调研、产品方向对齐、讨论输入  
> 适用范围：智能投放与分析 Agent 第一阶段方案设计；安全风控 Agent 作为第二能力线预留  
> 当前任务边界：只形成设计与讨论文档，不修改现有 Java 代码、数据库、网关配置或前端产物  
> 核心原则：先构建独立 Agent 能力，再通过现有 `admin` 管理链路增量接入短链项目；不推倒重构原项目  

---

## 0. 文档定位

本文档用于把外部成熟短链、深度链接、营销触达、安全扫描产品的能力拆解为本项目可落地的设计输入。

本项目当前已经具备短链创建、分组、跳转、访问统计、地域/设备/浏览器画像、回收站和后台管理能力。第一阶段不应把系统重构为完整广告平台，而应把现有能力包装成可被 Agent 使用的工具集合，让 Agent 承担以下任务：

```text
自然语言理解投放需求；
调用现有短链管理接口；
基于现有统计数据做确定性分析；
用大模型生成解释、建议、复盘摘要；
对所有写操作执行人工确认；
保留可追溯的工具调用、数据快照和回答依据。
```

## 1. 调研对象

### 1.1 Bitly

参考资料：

- Bitly Assist: https://bitly.com/pages/ai/bitly-assist
- Bitly AI Connections Platform: https://bitly.com/pages/ai
- Bitly LLM Integrations: https://bitly.com/pages/products/llm-integrations

可借鉴能力：

| 能力 | 外部做法 | 本项目落地点 |
|---|---|---|
| 对话式分析 | 用户直接用自然语言询问链接和二维码表现 | `AgentInsightService` 调用短链统计接口，输出趋势、异常和建议 |
| 对话式创建 | 基于用户描述生成链接、标题、二维码、标签 | Agent 解析活动、渠道、原始 URL、有效期、描述，调用现有创建接口 |
| 批量处理 | 一次生成多条链接或处理多个资产 | 复用 `batchCreateShortLink`，支持渠道列表批量生成 |
| 外部集成 | 让外部 LLM 或自动化工具创建可追踪短链 | 后续可暴露受控 Agent Tool API，而不是直接开放数据库 |
| 品牌与标签 | 用品牌域名、标签和 campaign metadata 管理资产 | 第一阶段先把 `gid` 作为活动容器，把 `describe` 作为轻量元数据 |

对本项目的启发：

```text
Agent 不只是聊天问答，而是“链接资产操作入口”。
回答必须绑定真实统计数据，不能让 LLM 自己编指标。
创建动作必须可预览、可确认、可追溯。
```

## 2. 外部方案共性总结

### 2.1 成熟产品不只做短链，而是做链接资产管理

成熟短链产品会围绕以下对象组织能力：

```text
Link；
Campaign；
Channel；
Domain；
QR Code；
UTM；
Audience；
Attribution；
Report；
Risk Review。
```

本项目第一阶段不新增全部对象，只做低侵入映射：

| 外部对象 | 第一阶段映射 | 后续正式对象 |
|---|---|---|
| Campaign | `gid` 分组 | `t_agent_campaign` |
| Channel | `describe` 中结构化片段 | `channel` 字段 |
| Link | `t_link` | 继续复用 |
| Report | Agent 生成结果，不持久或轻持久 | `t_agent_report` |
| Recommendation | Agent 回答中的建议 | `t_agent_recommendation` |
| Risk Review | 预留，不强制启用 | `t_url_risk_review` |

### 2.2 AI 主要用于理解、解释和编排，不替代事实源

成熟产品的 AI 能力通常围绕：

```text
自然语言入口；
自动生成链接/参数/标签；
统计数据摘要；
异常和机会识别；
报告生成；
操作建议。
```

对应本项目红线：

```text
LLM 不能直接生成指标；
LLM 不能越权读取用户分组；
LLM 不能直接删除或回收链接；
LLM 不能绕过确认创建批量短链；
LLM 回答必须引用工具返回的数据快照。
```

### 2.3 真正投放优化需要转化事件，但第一阶段可以先做点击智能

Branch、AppsFlyer 这类产品的价值在归因和转化，而本项目当前只有点击侧统计。因此第一阶段目标应该写清楚：

```text
支持点击表现分析；
支持访问画像解释；
支持活动短链创建；
支持低风险投放建议；
不宣称 ROI、CPA、GMV、支付转化率优化。
```

## 3. 建议第一阶段产品定位

产品名建议：

```text
短链智能投放与分析 Agent
ShortLink Campaign Analyst Agent
```

一句话定位：

```text
面向短链后台用户的智能运营助手，以现有分组、短链和统计接口为事实源，通过自然语言完成投放短链创建、批量生成、活动效果分析、异常发现和复盘建议，并通过人工确认保护所有写操作。
```

第一阶段主对象：

```text
AgentSession；
AgentMessage；
ToolCall；
PendingAction；
InsightReport；
ShortLink / Group / StatsSnapshot。
```

第一阶段不做：

```text
不改造跳转热路径；
不新增智能路由；
不做自动投放预算；
不做 ROI/CPA；
不做手机号级追踪；
不把安全风控 Agent 强制接入创建链路；
不替代原后台列表与统计页。
```

## 4. 方案路线讨论

### 4.1 Agent 放在哪里

| 方案 | 描述 | 优点 | 缺点 | 建议 |
|---|---|---|---|---|
| A. 嵌入 `admin` Java 模块 | 在现有后台直接新增 Agent Controller/Service | 接入快，部署简单 | Agent 运行、Prompt、Trace、评测会污染业务模块 | 不推荐作为长期形态 |
| B. 嵌入 `aggregation` 模块 | 聚合 admin/project 后新增 agent 包 | 本地演示方便 | 聚合模块本身更像启动器，边界不清 | 可作为临时 demo |
| C. 独立 Agent Service + Admin Adapter | 先构建独立 Agent 服务，通过 admin 受控工具接口调用现有能力 | 边界清晰、可测试、可替换模型、符合“先构建 Agent 再接入” | 多一个服务，需要配置和调用链 | 推荐 |

推荐结论：

```text
采用 C：独立 Agent Service + Admin Adapter。
Agent Service 先独立完成工具编排、Prompt、Trace、确认机制。
现有 Java 项目只新增轻量 Admin Adapter，负责鉴权、权限校验、工具执行和结果脱敏。
```

1. Agent Runtime 你更倾向于独立 Python 服务，还是 Java 内嵌服务？

## 5. 当前推荐讨论结论

我建议第一轮开发目标收敛为：

```text
先独立构建 ShortLink Campaign Analyst Agent。
Agent 只通过 Admin Adapter 调用现有短链能力。
第一阶段支持自然语言创建短链、批量创建短链、活动效果分析、异常解释和复盘建议。
所有写操作必须生成 PendingAction，用户确认后再执行。
安全风控 Agent 只做接口预留和文档设计，第二阶段再接外部 URL 扫描与风险审核。
```

这个路线最适合当前项目，因为它不要求重构 `project` 跳转链路，也不要求立刻修复全部历史问题，却能尽快做出真正有 Agent 感的能力。

# 风控生产编排与批处理可靠性 Implementation Plan

## 当前落地状态

截至 2026-07-10，本计划对应的后端生产编排能力已经完成实现，当前阶段以本地单元测试和 H2 MySQL mode 验证为准，不启动真实 DeepSeek、MySQL、Redis、admin 或 project 服务。

已落地的核心约束：

```text
1. 每个两小时窗口生成确定性 batchId，只允许一个未过期 owner lease 执行。
2. 画像、分组聚合和分析任务发布都必须携带当前 batch owner token。
3. 短链画像与分组画像按 batchId 幂等保存，禁止不同批次互相覆盖。
4. 单目标统计失败只记录到批次失败摘要，不中断同批其他短链。
5. 只有 SUCCEEDED/PARTIAL_SUCCESS 批次中的异常分组任务可以被 Worker 领取。
6. FAILED 或 lease 已过期的 RUNNING 批次按公平队列恢复，不阻塞当前窗口。
7. SecurityRisk Graph 使用结构化 analysisInput 精确消费指定 batch 和候选画像。
```

## 生产可靠性补充

### 1. 批次租约不可绕过

批处理写链路统一收口为：

```text
RiskProfileBatchCoordinator 获取 lease
  -> RiskProfileBatchService.runOnce(windowEnd, ownerToken)
  -> 短链画像/分组画像 Repository 校验 owned RUNNING batch
  -> RiskAnalysisJobPublicationService 在同一租约内发布分析任务
  -> JdbcRiskProfileBatchRepository.complete 以 owner token + 未过期 lease 完成批次
```

旧的无 owner 写入和恢复入口已经删除。测试必须通过 `RiskProfileTestFixture` 或真实 Repository 获取租约，不能伪造绕过批次 fence。

### 2. 失败批次恢复与公平轮转

`findRecoverableBefore` 只选择：

```text
window_end < 当前窗口结束时间
且 status = FAILED
或 status = RUNNING 且 lease_until 为空/已过期
```

候选按 `update_time, window_end, id` 排序，并受 `RISK_PROFILE_FAILED_RECOVERY_LIMIT` 限制。每次恢复前先调用 `recordRecoveryAttempt` 更新 `update_time`，即使批次窗口或 batchId 非法、恢复协调失败，也会进入队尾，避免坏记录长期占满 limit 饿死后续批次。

恢复查询使用：

```sql
idx_agent_risk_profile_batch_recovery
    (update_time, window_end, id, status, lease_until)
```

新库由 `agent_service_schema.sql` 创建；存量 MySQL 8 由生产迁移脚本显式 `ALTER TABLE ... ADD INDEX`。

### 3. Graph 证据状态

SecurityRisk Graph 不再把“查询成功但零记录”当作数据源故障：

```text
NOT_REQUESTED  未请求业务证据，普通对话可继续调用 LLM。
AVAILABLE      Tool 或画像返回可用证据。
NO_DATA        Tool 成功但数据为空，不调用 LLM，返回无风险证据提示。
SOURCE_FAILURE 请求的数据源真实失败，返回数据源不可用降级提示。
```

批处理模式下的 LLM 异常仍保持 fail-fast，让 Job Worker 进入可观测的重试状态；确定性画像、事件和策略逻辑不依赖 LLM 成功。

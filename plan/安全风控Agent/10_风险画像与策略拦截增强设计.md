# 风险画像与策略拦截增强设计

> **阶段定位：** 承接 `09_SecurityRiskAgent_E2E闭环计划.md`。当前 SecurityRisk Agent 已完成诊断闭环，本阶段设计下一轮后端增强：风险画像、风险事件、风险快照、人工审核、策略落库、Redis 策略同步和 Gateway 热路径拦截。

## 1. 设计目标

本阶段目标不是让大模型全量巡检所有短链，而是建立一套可重跑、可测试、可展示、可处置的风控后端体系：

```text
访问埋点与统计数据
  -> risk-profile 批处理生成短链画像
  -> risk-profile 聚合生成分组画像
  -> risk-detection 选出每组 Top 10 异常短链
  -> security-risk Agent 精准解释异常、生成处置建议
  -> risk-center 记录事件、快照、审核、审计
  -> risk-policy 同步 Redis 策略
  -> gateway 在短链跳转热路径拦截风险策略
```

核心约束：

```text
画像计算不依赖 LLM；
页面展示不走 Agent；
Agent 只处理异常候选和组级解释；
Gateway 只读 Redis，不查 MySQL，不调用 agent-service；
不保存原始 IP/user，使用 salted hash；
DISABLE_SHORT_LINK 不改 t_link.enable_status，使用风控策略拦截。
```

## 2. 已确认决策

| 主题 | 决策 |
|---|---|
| 目标维度 | GROUP 与 SHORT_LINK 都是一等风控对象 |
| 展示形态 | 分组展示组级风险卡片，并展示组内每条短链的最新风险卡片 |
| 保存策略 | 保存风险事件历史，同时维护最新风险快照 |
| 风险等级 | 三档：LOW、MEDIUM、HIGH |
| 人工审核动作 | CONFIRM_RISK、FALSE_POSITIVE、IGNORE、WATCH、UNWATCH |
| 风险数据归属 | agent-service 维护风险表；物理 MySQL 可与现有服务共用 |
| 风险查询 | 普通 API，不封装成 Agent Tool，不走 LLM |
| 诊断链路 | SecurityRisk Agent 读取画像、解释原因、生成处置建议、可调用工具处理短链 |
| 画像计算 | risk-profile 服务负责短链画像与分组画像，纯规则/统计，可测试、可重跑 |
| 批处理频率 | 每 2 小时更新一次 |
| 扫描范围 | 近 7 天有访问的短链 |
| 画像窗口 | 最近 2 小时、过去 24 小时、过去 7 天 |
| Agent 深度处理 | 每个分组每批最多 Top 10 异常短链 |
| 自动执行 | LIMIT_RATE 可条件化自动执行；DISABLE_SHORT_LINK、BLOCK_IP、LIMIT_TIME_WINDOW 必须人工确认 |
| Gateway | 补充可验证路由基线，新增 RiskPolicyGatewayFilter |
| Redis key | 使用标准化 domain + shortUri |
| Domain 标准化 | 80/443 去端口，非默认端口保留 |
| IP 来源 | trusted-proxy-enabled=true 时优先 X-Forwarded-For 第一个 IP，其次 X-Real-IP，最后 remoteAddress |

## 3. 总体架构

```text
agent-service
  securityriskagent/
    读取画像，解释风险，生成处置建议，写风险事件与快照

  riskprofile/
    计算 ShortLinkRiskProfile
    聚合 GroupRiskProfile
    每 2 小时定时批处理

  riskdetection/
    根据画像计算 anomalyScore
    生成 reasonCodes
    每组筛选 Top 10 异常短链

  riskcenter/
    普通风险查询 API
    风险事件历史
    最新风险快照
    人工审核记录
    动作审计

  riskpolicy/
    风控策略持久化
    Redis 策略同步
    策略失效与撤销

gateway
  RiskPolicyGatewayFilter
    短链跳转前查 Redis
    命中策略则拦截
    未命中继续转发 project

project
  保持原跳转逻辑
  不直接依赖 agent-service 风控表
```

### 3.1 画像线

```text
近 7 天有访问短链
  -> ShortLinkRiskProfileJob
  -> ShortLinkRiskProfile
  -> ShortLinkRiskDetector
  -> ShortLinkRiskSnapshot
  -> GroupRiskProfileAggregator
  -> GroupRiskProfile
```

画像线只做确定性统计和规则评分，不调用 LLM。它是风险展示、异常候选和后续 Agent 分析的事实基础。

### 3.2 Agent 线

```text
Top 10 异常短链画像
  -> SecurityRisk Agent
  -> 风险解释
  -> 证据摘要
  -> 处置建议
  -> RiskEvent
  -> RiskSnapshot
  -> RiskActionProposal
```

Agent 不全量扫描分组内所有短链，只处理风险画像筛出的异常候选。组级 Agent 读取分组画像和 Top 10 短链画像，生成组级风险总结和建议。

### 3.3 展示查询线

```text
admin 分组页
  -> admin 风险查询 API
  -> agent-service risk-center API
  -> t_agent_risk_snapshot / t_agent_risk_event / t_agent_group_risk_profile
```

展示查询不走 Agent，不走 Tool，不走 LLM。

### 3.4 策略拦截线

```text
审核确认或自动限流
  -> agent-service risk-policy
  -> 写 MySQL 策略表
  -> 写 Redis 生效策略
  -> gateway RiskPolicyGatewayFilter
  -> 命中后直接拦截
```

Gateway 是热路径读取方，只读 Redis，不调用 agent-service。

### 4.2 分组画像

分组画像基于同批短链画像聚合，不直接扫描访问明细。

```text

- profileWindowStart

- totalShortLinksScanned

- highRiskCount

- disabledCount
- avgRiskScore
- maxRiskScore
- topRiskShortLinks[10]
- groupReasonCodes
- groupRiskLevel
- groupRiskScore
- riskTrend7d
- agentSummary
```

`riskTrend7d` 使用日粒度：

```json
[
  {
    "date": "2026-07-03",
    "avgRiskScore": 18,
    "maxRiskScore": 64,
    "mediumRiskCount": 3,
    "highRiskCount": 1
  }
]
```

`riskTrend7d` 必须由历史画像或风险事件聚合生成，Agent 不得编造趋势。

### 4.3 批处理规则

```text
频率：每 2 小时执行一次
扫描范围：近 7 天有访问的短链
计算窗口：最近 2 小时、过去 24 小时、过去 7 天
分组策略：按 gid 分批处理
Agent 成本控制：每个 gid 每批最多 Top 10 异常短链进入 Agent 深度分析
```

观察中短链如果近 7 天没有访问，保留观察状态，但本轮不重算访问画像。

## 7. 敏感数据与 Hash 策略

### 7.1 存储原则

```text
画像表、风险事件表、快照表不保存原始 IP。
画像表、风险事件表、快照表不保存原始 user / visitor 标识。
如需匹配，保存 ipHash / visitorHash / userHash。
LLM prompt、Agent response、checkpoint 不展示原始 IP/user，也不展示 hash 明细。
```

### 7.2 Hash 配置

```yaml
short-link:
  risk:
    hash-salt: ${RISK_HASH_SALT:}
```

真实 salt 不提交到仓库。缺少 salt 时，生产环境必须拒绝启用 BLOCK_IP 策略与 hash 明细持久化；本地测试可以使用测试专用 salt。

### 7.3 Gateway IP 来源

```text
trusted-proxy-enabled=true：
  1. X-Forwarded-For 第一个 IP
  2. X-Real-IP
  3. remoteAddress

trusted-proxy-enabled=false：
  只使用 remoteAddress
```

配置：

```yaml
short-link:
  risk:
    policy:
      trusted-proxy-enabled: true
```

生产是否信任 `X-Forwarded-For` 取决于 nginx 是否清洗并覆盖该请求头。

## 8. 处置动作设计

### 8.1 动作分类

自动允许：

```text
WATCH
UNWATCH
ADD_NOTE
CREATE_REVIEW
LIMIT_RATE 条件化自动
```

必须人工确认：

```text
DISABLE_SHORT_LINK
BLOCK_IP
LIMIT_TIME_WINDOW
```

第一版不提供不可逆删除动作。短链风控禁用使用 `DISABLE_SHORT_LINK`，不使用 `DELETE_SHORT_LINK`。

### 8.2 自动限流条件

`LIMIT_RATE` 允许 Agent 自动执行，但必须满足确定性条件：

```text
riskLevel = HIGH
riskScore >= 80
reasonCodes 至少命中以下任意两个：
  - TRAFFIC_SPIKE
  - IP_CONCENTRATION
  - HIGH_REPEAT_VISIT
  - PEAK_HOUR_BURST
```

不满足条件时，Agent 只生成 pendingAction，不自动执行。

### 8.3 DISABLE_SHORT_LINK

`DISABLE_SHORT_LINK` 第一版真实生效，但不修改 `t_link.enable_status`，也不复用回收站语义。它通过风控策略写 Redis，由 Gateway 拦截短链跳转。

对访问用户表现：

```text
DISABLE_SHORT_LINK -> 404 或 /page/notfound
```

不向访问用户暴露风控原因。

### 8.4 BLOCK_IP、LIMIT_RATE、LIMIT_TIME_WINDOW

```text
BLOCK_IP：
  第一版策略落库、Redis 生效、Gateway 可拦截。

LIMIT_RATE：
  第一版策略落库、Redis 生效、Gateway 按 IP + 短链维度限流。

LIMIT_TIME_WINDOW：
  第一版策略落库、Redis 生效、Gateway 按允许时间窗判断。
```

## 9. Redis 策略设计

### 9.1 Domain 标准化

```text
Host: nurl.ink              -> nurl.ink
Host: nurl.ink:80           -> nurl.ink
Host: nurl.ink:443          -> nurl.ink
Host: 127.0.0.1:8000        -> 127.0.0.1:8000
Host: localhost:5174        -> localhost:5174
```

### 9.2 短链策略 key

```text
risk:policy:short-link:disable:{domain}:{shortUri}
risk:policy:short-link:rate-limit:{domain}:{shortUri}
risk:policy:short-link:time-window:{domain}:{shortUri}
```

### 9.3 IP 策略 key

```text
risk:policy:ip:block:{ipHash}
```

### 9.4 限流计数 key

`LIMIT_RATE` 第一版按 IP + 短链粒度：

```text
risk:rate:{domain}:{shortUri}:{ipHash}
```

策略 payload：

```json
{
  "limit": 60,
  "windowSeconds": 60,
  "action": "LIMIT_RATE"
}
```

Gateway 执行逻辑：

```text
1. 查 risk:policy:short-link:rate-limit:{domain}:{shortUri}
2. 如果存在，计算 ipHash
3. INCR risk:rate:{domain}:{shortUri}:{ipHash}
4. 首次计数设置 TTL = windowSeconds
5. count > limit 时返回 429
```

### 9.5 限时访问策略

策略结构支持 `allowedWindows` 和 `blockedWindows`，第一版只开放 `allowedWindows`。

```json
{
  "action": "LIMIT_TIME_WINDOW",
  "timezone": "Asia/Shanghai",
  "allowedWindows": ["09:00-22:00"],
  "blockedWindows": []
}
```

执行语义：

```text
allowedWindows 非空：
  当前时间在任一 allowedWindow 内 -> 放行
  否则 -> 403
```

## 10. Gateway 路由与 Filter 设计

### 10.1 当前仓库状态

当前 `gateway` 模块已经存在：

```text
端口：8000
技术：Spring Cloud Gateway
Redis：StringRedisTemplate
已有过滤器：TokenValidateGatewayFilterFactory
```

但仓库内没有可验证的 `spring.cloud.gateway.routes` 配置，也没有 discovery locator 配置。因此当前仓库只能证明 gateway 可编译，不能证明短链跳转已经经过 gateway。

### 10.2 路由基线

需要补充 dev/local 可验证路由。生产可以继续使用 Nacos 外部路由，但仓库必须保留可测试基线。

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: short-link-admin-api
          uri: lb://short-link-admin
          predicates:
            - Path=/api/short-link/admin/v1/**
          filters:
            - TokenValidate

        - id: short-link-project-api
          uri: lb://short-link-project
          predicates:
            - Path=/api/short-link/v1/**

        - id: short-link-redirect
          uri: lb://short-link-project
          predicates:
            - Path=/{shortUri}
          filters:
            - RiskPolicy
```

路由顺序必须保证 `/api/**` 优先于 `/{shortUri}`，避免 `/api` 被当作短链后缀。

### 10.3 RiskPolicyGatewayFilter

新增：

```text
gateway/src/main/java/com/nageoffer/shortlink/gateway/filter/RiskPolicyGatewayFilterFactory.java
```

职责：

```text
1. 仅挂在短链跳转路由。
2. 从 Host 和 path 解析 domain + shortUri。
3. 标准化 domain。
4. 查 Redis disable / time-window / rate-limit / block-ip 策略。
5. 命中策略时直接返回。
6. 未命中时继续 chain.filter。
```

响应规则：

```text
DISABLE_SHORT_LINK -> 404 或 /page/notfound
BLOCK_IP -> 403
LIMIT_RATE -> 429
LIMIT_TIME_WINDOW -> 403
```

第一版建议直接返回对应 HTTP 状态，是否跳转 `/page/notfound` 可在实施阶段结合现有静态页面决定。

## 11. API 设计

### 11.1 风险查询 API

风险查询是普通 API，不是 Agent Tool。

```text
GET /internal/short-link-agent/v1/risk/groups/{gid}/overview
GET /internal/short-link-agent/v1/risk/groups/{gid}/short-links
GET /internal/short-link-agent/v1/risk/short-links/{domain}/{shortUri}
GET /internal/short-link-agent/v1/risk/events
```

这些接口供 admin 后端调用，admin 再对前端暴露正式 API。

### 11.2 审核 API

```text
POST /internal/short-link-agent/v1/risk/reviews
POST /internal/short-link-agent/v1/risk/policies/{policyId}/disable
```

审核 API 需要记录 reviewer、reviewAction、reviewNote 和 traceId。

### 11.3 策略执行 API

策略执行仍在 agent-service 内部，由审核服务或 Agent 自动动作调用，不直接暴露给公网。

```text
RiskPolicyService.activatePolicy(...)
RiskPolicyService.disablePolicy(...)
RiskPolicyRedisPublisher.publish(...)
RiskPolicyRedisPublisher.revoke(...)
```

## 13. 验收标准

```text
1. gateway 仓库内存在可测试路由基线。
2. /{shortUri} 路由命中 RiskPolicyGatewayFilter。
3. /api/** 路由不误走 RiskPolicyGatewayFilter。
4. Redis disable 策略命中时短链跳转返回 404 或 notfound。
5. Redis block-ip 策略命中时返回 403。
6. Redis rate-limit 策略超过阈值时返回 429。
7. Redis time-window 策略窗口外返回 403。
8. risk-profile 每 2 小时批处理近 7 天有访问短链。
9. 短链画像包含 traffic、concentration、behavior、risk 指标。
10. 分组画像包含 riskTrend7d 和 Top 10 异常短链。
11. SecurityRisk Agent 不全量扫描短链，只处理 Top 10 异常候选。
12. 风险查询 API 不走 Agent、不走 LLM。
13. 原始 IP/user 不进入画像表、事件表、快照表、LLM prompt、响应和 checkpoint。
14. LIMIT_RATE 自动执行必须满足 HIGH、riskScore>=80、强 reasonCode 至少两个。
15. DISABLE_SHORT_LINK、BLOCK_IP、LIMIT_TIME_WINDOW 必须人工确认。
```

## 14. 待实施时再次核对的问题

以下问题不影响设计确认，但实施前需要用代码和环境再次核对：

```text
1. 生产 Nacos 中 gateway 路由是否已经覆盖短链跳转域名。
2. nginx 是否清洗并覆盖 X-Forwarded-For。
3. project 访问统计是否能直接支撑 topVisitorShare、repeatVisitRatio、topDeviceShare、topBrowserShare。
4. /page/notfound 是否由 gateway 可直接 redirect，还是需要返回 404 让上游处理。
5. RISK_HASH_SALT 在本地、测试、生产环境的注入方式。
```

# 安全风控 Agent 外部方案调研

## 1. 调研目标

安全风控 Agent 的目标不是第一版就替代 WAF、Bot 管理或完整反欺诈平台，而是把大厂风控产品中的通用思路落到短链接项目已有数据上：

```text
风险评分
风险标签
证据归因
分级处置
人工确认
隐私最小化
```

## 2. 参考方案

### 2.1 Cloudflare Bot Management

Cloudflare Bot Management 的核心启发是 **Bot Score**：请求被归一化成风险分数，并结合多类检测信号进入规则动作。对短链接项目而言，可以把聚合统计和访问明细转成 `riskScore`，再由 Agent 输出原因和建议。

参考链接：[Cloudflare Bot Score](https://developers.cloudflare.com/bots/concepts/bot-score/)

可借鉴点：

```text
使用 0-100 风险分数，而不是只输出“正常/异常”。
把机器判断和规则处置解耦。
证据需要能回看，例如 Top IP、PV/UV、小时集中度。
```

### 2.2 Google reCAPTCHA Enterprise

reCAPTCHA Enterprise 的 assessment 会返回 score 和 risk reasons。它的启发是：模型输出必须和可解释的原因码绑定，否则运营人员无法判断是否应该处置。

参考链接：[reCAPTCHA Enterprise 评估解释](https://cloud.google.com/recaptcha/docs/interpret-assessment-website)

可借鉴点：

```text
输出 riskScore + reasonCodes。
原因码用于 UI 展示、审计、规则调优和后续告警。
低置信度时只建议观察，不做强动作。
```

### 2.3 AWS WAF Bot Control / Fraud Control

AWS WAF 的托管规则会把请求打上 labels，再由用户规则决定阻断、挑战或放行。它的启发是：第一阶段不需要做完所有处置能力，先把风险信号结构化。

参考链接：[AWS WAF Bot Control](https://docs.aws.amazon.com/waf/latest/developerguide/aws-managed-rule-groups-bot.html)

可借鉴点：

```text
风险信号标签化，例如 ip_concentration、hour_burst。
动作策略分层，例如 observe、review、challenge、block。
自动动作要有明确阈值和审计。
```

### 2.4 OWASP Automated Threats

OWASP Automated Threats 提供自动化攻击类型分类。短链接项目当前不直接处理登录撞库、支付欺诈等复杂场景，但短链跳转天然容易遇到刷量、爬虫、恶意传播、批量访问和异常来源集中。

参考链接：[OWASP Automated Threats](https://owasp.org/www-project-automated-threats-to-web-applications/)

可借鉴点：

```text
把风险场景命名清楚。
不要把所有异常都叫“攻击”。
风控结论需要表达置信度和可能误伤。
```

## 3. 对本项目的落地结论

第一阶段采用以下风控输出合同：

```text
riskScore：0-100，越高越危险
riskLevel：low / medium / high / critical
reasonCode：稳定原因码
evidence：脱敏证据
recommendations：建议动作
pendingActions：需要用户确认的动作草案
```

第一阶段不做：

```text
不自动封禁 IP。
不自动删除短链。
不自动冻结分组。
不直接写 Redis 黑名单。
不绕过 admin 权限执行任何处置。
```
